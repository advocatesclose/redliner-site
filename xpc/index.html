Process Isolation

Three services.
Three processes.
Zero shared memory.

Every heavyweight operation in Redliner runs outside the main application process. Each XPC service lives in its own sandbox, with its own address space, its own lifecycle, and its own failure domain. A crash in the conversion engine cannot reach your document. That is not a feature. It is the architecture.

No document content crosses the process boundary. The only thing on the wire is a file path.

Pointers, not payloads. By design.

How It Works

Technical Deep Dive

The proxy pattern,
the lifecycle, the wire.

XPC is Apple's inter-process communication framework. Each XPC service is a separate binary, bundled inside the application, launched on demand by launchd. It runs in its own process with its own address space. When the last connection to the service is invalidated, launchd is free to terminate it. When the next connection is opened, launchd launches it again.

On the service side, an NSXPCListener waits for incoming connections. When a connection arrives, the listener delegate decides whether to accept it. If accepted, it creates a handler object, sets it as the connection's exportedObject, and declares the protocol the handler conforms to via exportedInterface. In Redliner, each service delegate follows this same pattern: DownloadServiceDelegate, PandocServiceDelegate, and RedlineServiceDelegate each implement NSXPCListenerDelegate with the same structure.

On the app side, clients like PandocClient and RedlineClient create an NSXPCConnection with the service's bundle identifier. They set a remoteObjectInterface declaring the expected protocol, resume the connection, and obtain a proxy. The proxy object looks and behaves like a local instance of the protocol. When you call a method on it, the XPC runtime serializes the arguments, sends them across the Mach port to the service process, deserializes them, and calls the real method on the handler. The reply block crosses the boundary in the opposite direction.

Redliner wraps every XPC call in Swift's withCheckedThrowingContinuation, bridging the Objective-C reply-block pattern into structured async/await. The connection is created, the call is made, and the connection is invalidated in the reply handler. Each call gets its own connection. This is deliberate: it keeps lifecycle simple, avoids stale connection state, and lets launchd manage process termination cleanly.

File coordination is central to the design. The RedlineServiceHandler uses NSFileCoordinator.prepare(forReadingItemsAt:writingItemsAt:error:) to coordinate two reads and one write in a single atomic batch. This is Apple's recommended approach when multiple coordinated accesses are needed. It prevents deadlock and ensures no other process modifies the files mid-operation.

The DownloadService and PandocService both implement a 5-minute idle shutdown timer. When the last connection invalidates, a DispatchSourceTimer starts counting. If no new connection arrives within 300 seconds, the service calls exit(0). macOS will relaunch it transparently on the next XPC call. The result is a system where services exist only when needed, consume no resources when idle, and restart without the app noticing.

For batch operations, RedlineClient.batchRedline(jobs:) uses a Swift ThrowingTaskGroup to dispatch multiple redline jobs in parallel. Each job creates its own NSXPCConnection, its own handler in the service process, and its own coordinated file access. The task group collects results as they complete. Ten redlines can run simultaneously, each fully isolated.

API Surface

API
surface.
NSXPCConnection
Created per call with NSXPCConnection(serviceName:). Sets remoteObjectInterface, resumes, obtains proxy via remoteObjectProxyWithErrorHandler. Invalidated in the reply block.
NSXPCListener
Each service calls NSXPCListener.service() in its main.swift. The listener delegate accepts connections and vends handler objects. Runs the main run loop indefinitely.
NSFileCoordinator
Used by RedlineServiceHandler with prepare(forReadingItemsAt:writingItemsAt:error:). Coordinates two reads and one write atomically. Prevents cross-process file conflicts.
DispatchSourceTimer
5-minute idle shutdown in DownloadService and PandocService. Fires exit(0) after 300 seconds of inactivity. Reset on every new connection and every invalidation.
Protocol Signatures
DownloadServiceProtocol, PandocServiceProtocol, RedlineServiceProtocol — all @objc public protocol types shared between app and service targets. Reply blocks return file paths or data plus optional NSError.
ThrowingTaskGroup
RedlineClient.batchRedline(jobs:) dispatches parallel redline jobs using withThrowingTaskGroup. Each job gets its own NSXPCConnection. Results collected in completion order.

Privacy by Process Isolation

Separate processes. Separate memory. No exceptions.

Each XPC service runs in its own process with its own virtual address space. One service cannot read another service's memory. The main application cannot read the service's memory. No document content is serialized over the XPC connection — the services receive file paths, read from the filesystem, write to the filesystem, and return file paths. Every byte of your work stays on your machine. There is no server. There are no API calls. There is no telemetry. The architecture makes data exfiltration structurally impossible — not just prohibited by policy, but prevented by the process model itself.
