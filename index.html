Thesis

On Complexity

Programmers have solved
problems that other fields
have not.

Computer programmers as we know them today have existed for less than a century. They are not credentialed or licensed. They do not sit an exam. And yet the things entrusted to computer programs are not somehow less important than the things entrusted to professionals who do. A Boeing 747 does not stay aloft because it is required by law to do so. It stays aloft because it is governed by computer programs that make it able to do so.

How this is possible is worth understanding. Over a remarkably short period, programmers have developed an astonishing array of tools, practices, protocols, and processes for managing complexity and building large-scale systems that interoperate with one another. iPhones are not less useful when three billion people have one. The tools do not degrade with scale.

Laws, by contrast, become harder to use when the number of people who must apply them increases. It is not obvious that this limitation reflects some fundamental constraint. It may simply reflect a gap in tooling.

Redliner is an attempt to close that gap. If you print a document to Word and then print it to PDF, you have not done one thing twice. You have done two things once each. Printing is printing. Redlining is redlining. The conventional approach asks you to repeat yourself — to manually produce each output, each comparison, each format. We think this is unnecessary. A single edit should propagate to every output as a matter of architecture, not effort.

Architecture
01

XPC Services

Document conversion and redline generation run in dedicated XPC services — separate processes with independent memory spaces, managed by launchd, created and destroyed on demand. Each service does one thing. The app coordinates.

Multi-Process
02

Parallel Redlining

The redline engine implements a word-level Myers diff. Multiple redlines execute in parallel — each with its own NSXPCConnection, its own handler, its own coordinated file access via NSFileCoordinator.

Structured Concurrency
03

Zero-Copy IPC

No document content crosses the XPC boundary. The app passes file paths; the services read, process, and write directly to the filesystem. The main process never holds conversion artifacts in memory.

Privacy by Architecture

Privacy is not a trade-off. It is a design constraint.

XPC services run in separate processes with separate memory. No document content crosses the process boundary — only file paths. Edit transactions are encrypted with AES-GCM before they touch disk. The architecture makes the wrong thing hard to do, not merely prohibited.
