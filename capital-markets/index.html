EDGAR Import

Paste a URL from any filing on EDGAR. Redliner derives the accession number from the directory structure (the 18-digit format like 000149315224044245), reformats it with dashes (10-2-6), and fetches the filing index page to discover every exhibit. HTML Parsing

SECHTMLParser uses a compiled NSRegularExpression with .dotMatchesLineSeparators to extract every paragraph from SEC HTML. It reads the attribute string of each <p> tag -- checking for class names like H1, H2, Article, and Section -- to classify block type. Bullet points are detected via &#8226; entities and literal bullet characters. Internal tags are stripped. Entity artifacts are cleaned.

Deep Dive

Technical Essay

Why this data is
hard -- and interesting.

EDGAR filings are not web pages and they are not database rows. They are legal documents serialized as HTML -- semi-structured data with internal organization expressed through headings, exhibits, cross-references, numbered sections, and inline XBRL tags, all rendered via inconsistent markup that varies by filing agent, era, and document type.

A typical SEC filing uses deeply nested tables for layout, not CSS. Headings are not <h1> through <h6> tags -- they are <p> tags with class names like "H1", "H2", "Article", or "Section" embedded in an "attribute soup" alongside inline styles for margins, fonts, and spacing. A paragraph might carry class="H1" style="margin:0; font-family:Times" and the only way to know it is a heading is to scan the attribute string for the class name.

Bullet points do not use <ul> or <li>. They use <p> tags containing the HTML entity &#8226; or a literal Unicode bullet character. The parser must detect both forms and strip the bullet from the content after reclassifying the block type.

Then there is inline XBRL. Modern SEC filings wrap financial data in <ix:nonFraction>, <ix:nonNumeric>, and similar tags that interleave with the HTML content. These tags carry metadata like ix:name that identifies the XBRL concept. The parser must preserve this metadata while stripping the tags to extract clean text. The SECBlock model stores this in an optional [String: String] metadata dictionary.

Redliner's approach is deliberate: a single compiled NSRegularExpression with the pattern <p\b([^>]*)>(.*?)</p>, using .dotMatchesLineSeparators so content spanning multiple lines is captured correctly. Group 1 captures the attribute soup. Group 2 captures the raw content. The \b word boundary prevents false matches on <pre> or <param> tags. The regex is compiled once at load time and reused for every parse call.

After extraction, a tag-stripping pass removes internal HTML (<span>, <b>, <i>) and a cleanup pass normalizes non-breaking spaces (&nbsp;, &#160;). The result is a clean SECDocument with typed, structured blocks that can be rendered, diffed, or edited.

Mathematical Depth

Computational Methods

Why semi-structured data
requires serious math.

Parsing EDGAR is not a string-matching problem. The corpus is too large, too inconsistent, and too structurally varied for pattern-matching alone. Training on and generalizing across this material touches real computational depth.

The structural variation across filings is topological in nature -- document sections nest, cross-reference, and recombine in ways that resist tree representations and require reasoning about manifolds and connectedness. Spectral methods apply when decomposing the recurring structural patterns across thousands of filings: eigenvector analysis of section co-occurrence, Chebyshev polynomial approximations for smoothing noisy classification signals, and Fisher information matrices for quantifying how much each structural feature tells you about document type.

Time series methods matter because the EDGAR corpus is not stationary -- filing conventions drift. The HTML output of Donnelley, Toppan Merrill, and other filing agents has changed across decades. A parser that works on 2024 S-1 filings may fail on 2008 10-Ks. Modeling this requires non-stationary time series analysis, drift terms, and innovation matrices that capture how formatting conventions evolve. Convergence guarantees for parsing heuristics require analysis of overdetermined systems where multiple classification signals (class names, nesting depth, font attributes, position) must be reconciled.

Generating functions formalize the combinatorics of exhibit discovery -- how many distinct structural patterns exist in the filing index, and what is the probability that a new filing introduces a pattern outside the known set. Hessians of the loss surface characterize the stability of learned parsers under distribution shift. This is not NLP applied to legal text. It is infrastructure for one of the most interesting semi-structured datasets available to anyone with a network connection.

Technical Specs
The types
behind the
pipeline.
SECDocument
Codable, Sendable struct. Contains an array of SECBlock values. Converts to/from JSON. The canonical representation of a parsed filing.
SECBlock
UUID-identified block with SECBlockType enum (heading1, heading2, heading3, paragraph, definition, listMarker, listItem, unknown), content string, and optional XBRL metadata dictionary.
EDGARImporter
Async static API. importFiling(from:progress:) takes a URL and a progress callback. Returns a FilingBlueprint with sections split by markdown heading.
DownloadServiceProtocol
@objc XPC protocol. Two methods: download(from:to:reply:) for single documents, downloadFiling(from:to:reply:) for full filing index downloads.
NSRegularExpression
Compiled once at load time with .dotMatchesLineSeparators and .caseInsensitive. Pattern: <p\b([^>]*)>(.*?)</p>. Used for every parse call.
PandocClient
Converts downloaded HTML to markdown for section splitting and diffing. SEC headings become clean # and ## markers, making structural extraction trivial.
FilingBlueprint
The import result. Contains BlueprintSection arrays for Registration Statement, Exhibits, and Financial Statements. Each section holds BlueprintExhibit values with imported markdown or HTML.
NSDataDetector
Used for broad URL discovery in filing HTML. Combined with regex href extraction to catch both absolute and relative links. Deduplicates by filename.

Direct to Your Machine

Filings download directly to your machine via XPC. No intermediary server.

The DownloadService is a dedicated XPC process that fetches filings straight from SEC EDGAR to your local filesystem. It sets a SEC-compliant User-Agent header (Redliner/1.0 (Advocates Close Corp.)), strips EDGAR navigation chrome, and writes clean HTML to disk. The main application never touches the network. No proxy. No intermediary. No cloud. The filing goes from EDGAR to your ~/Library/Application Support/Redliner/Filings/ directory and nowhere else. The XPC service auto-terminates after five minutes of inactivity. macOS relaunches it on the next call.
