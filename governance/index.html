The Dataset

The Source Material

A public corpus with
hidden structure.

Every public company files governance documents with the SEC. Certificates of incorporation, bylaws, committee charters, codes of ethics, warrant agreements — all attached as exhibits to registration statements and annual reports. All freely downloadable via EDGAR.

These are not random text. They follow conventions. Article numbering schemes. Defined term patterns. Cross-reference structures. Boilerplate paragraphs that recur across thousands of filings with small but meaningful variations — a different threshold here, an added carve-out there. The same document type from two different law firms will share 80% of its language and differ in precisely the places that matter.

This makes governance documents a uniquely interesting dataset for structured data engineering. They sit in the gap between unstructured text and clean relational data. Too patterned to treat as blobs. Too variable to treat as fixed schemas. The engineering challenge is to extract the latent structure — to infer the schema that generated these documents, and to build tooling that operates on that schema rather than on raw text.

Redliner treats this corpus as a first-class data engineering problem. The filing matrix cross-references documents across filings. The schema pipeline parses XSD taxonomies into typed fields. The template engine renders structured output from schema-driven forms. Every layer operates on declared structure, not discovered patterns.

The Filing Matrix

FilingMatrixView

Every document. Every filing.
One cross-reference grid.

The filing matrix organizes documents by type, status, and entity. Rows are exhibit titles. Columns are filings. Cells show presence and diff status at a glance.

Each FilingBlueprint contains an array of BlueprintSection objects, each holding an array of BlueprintExhibit entries. The matrix extracts every unique exhibit title across all filings and normalizes them, stripping prefixes like "Form of" and "Amended and Restated" so that matching documents align regardless of naming convention.

Real fields from FilingData.swift drive the data: filerName, domicile, sic, ein, registrationNumber, executiveName, serviceUponName. Lawyer data is structured as an array of Lawyer objects, each carrying name1, name2, firm, and a full PostalAddress with street1, street2, city, state, zip, phone.

When two exhibits share a title but differ in content, the matrix shows a diff indicator. Click it and the system passes both versions to the comparison view. When they match, a checkmark. When absent, a dash. No ambiguity.

The Schema Pipeline

Deep Dive

XSD to rendered form.
Four stages. One pipeline.

Every governance document in Redliner flows through the same pipeline. An XSD taxonomy defines the schema. The parser extracts sections and fields. The engine generates forms. User input is captured and validated against the schema in real time.

XSDParser reads the raw XML and walks xs:element nodes, extracting the name, type, abstract, nillable, xbrli:periodType, and substitutionGroup attributes. Each element becomes a SchemaField. Role types with link:definition become SchemaSection objects, each mapped to an S-K item number like 1602 or 1603.

The resulting DocumentSchema holds sorted sections and typed fields. It exposes fillableFields (non-abstract, user-editable), dimensions (axis fields for table breakdowns), and tables (hypercube elements). Fields are grouped by inferred section prefix: Forepart, Prospectus Summary, Dilution, Sponsor, Trust Account, Redemption, De-SPAC Transaction.

FilledDocument captures user input against this schema. It stores values as [String: String], provides typed accessors for Bool, Decimal, and Int, and validates every entry against the schema's type and nillability constraints. Unfilled required fields surface as ValidationIssue.missingRequired. Type mismatches surface as ValidationIssue.typeMismatch. The render method replaces {{fieldName}} placeholders and marks anything unfilled as [MISSING: fieldName].

Mathematical Depth

Schema Inference

The mathematics of
generalizing across documents.

Inferring schemas from semi-structured documents is not a string-matching problem. It is a mathematical one. The corpus contains thousands of instances of the same document types, each a noisy realization of an underlying template. Recovering that template — and characterizing how instances vary around it — requires real mathematics.

A certificate of incorporation is not a fixed form. It is a point in a high-dimensional space of possible certificates, parameterized by jurisdiction, share structure, protective provisions, amendment thresholds, and dozens of other axes. Two certificates from the same law firm will be nearby in this space. Two from different firms will be further apart but still recognizably on the same manifold.

The engineering question is: given N observed documents of a type, what is the minimal schema that explains them? This is an overdetermined system — more observed constraints (provisions, clauses, structural patterns) than free parameters in the schema. The solution is not exact. It is an approximation problem, and the right tools come from numerical analysis, spectral theory, and information geometry.

Redliner's schema pipeline is the foundation for this work. The XSD parser, the typed fields, the template engine — these are the concrete infrastructure. The mathematical framework below describes the harder problem they are built to support.

Topology
Document structure as graphs
Articles, sections, subsections, cross-references, and defined terms form a directed graph. Topological invariants — connectivity, cycle structure, depth — distinguish document types and detect structural anomalies across filings.
Manifolds
Continuous variation in boilerplate
Boilerplate is not identical across filings. It varies continuously — a threshold changes from 50% to 66 2/3%, a carve-out is added or removed. The space of observed variations forms a low-dimensional manifold embedded in the full text space.
Spectral Methods
Clustering similar provisions
Given a similarity matrix over provisions (e.g., redemption clauses across 500 SPAC filings), spectral decomposition of the graph Laplacian reveals natural clusters — groups of provisions that are functionally equivalent despite surface-level variation.
Generating Functions
Template composition
A document template is a function from parameter values to rendered text. Composing templates — nesting a share-structure block inside a certificate template inside a filing template — is function composition. Generating functions encode the combinatorics of valid compositions.
Chebyshev Polynomials
Approximating field distributions
Numerical fields — authorized shares, par values, offering sizes — have empirical distributions across the corpus. Chebyshev polynomial approximation provides optimal uniform approximations of these distributions, enabling range validation and outlier detection with minimal terms.
Fisher Information
Measuring parameter sensitivity
Some schema parameters matter more than others. The Fisher information matrix quantifies how sensitively the distribution of observed documents depends on each parameter — identifying which fields carry the most structural information and which are noise.

More constraints than degrees of freedom. An overdetermined system. The schema is the least-squares solution.

Technical Specs
The types
behind the
documents.
FilingData
SwiftData @Model with CoverPageData, PostalAddress, and Lawyer value types. Converts to flat [String: TemplateValue] dictionaries for template rendering. Fields: filerName, domicile, sic, ein, registrationNumber, executiveName.
FilingBlueprint
Defines filing structure: name, lawyerName, sections: [BlueprintSection], defaultFields: [String: String]. BlueprintSection holds exhibits with optional templateKey and importedMarkdown. LawyerProfile presets generate complete structures per firm.
SchemaField
Parsed from xs:element. Properties: id, name, label, type: FieldType, isAbstract, isNillable, periodType, substitutionGroup. Computed: isFillable, isDimension, isTable, isMember. Labels humanized from camelCase via SchemaField.humanize().
SchemaSection
Disclosure section from XSD link:roleType. Fields: id, roleURI, title, skReference, sortKey. Parses definitions like "99554.2 - Disclosure - S-K 1602, SPAC Registered Offerings" into structured components. Sorted by sortKey for deterministic ordering.
TemplateEngine
Renders markdown templates with TemplateContext scope stack. Supports {{var}}, {{var | filter}}, {{#if}}/{{#unless}}, {{#each}}, {{#with}}, {{> partial}}, {{compute expr}}, {{format value "style"}}. Values are TemplateValue enum: .string, .number, .bool, .date, .array, .dictionary.
FilledDocument
Runtime document instance: schema: DocumentSchema, values: [String: String], metadata: DocumentMetadata. Typed accessors for Bool, Decimal, Int. Full validation against schema constraints. Template rendering with [MISSING] markers for unfilled placeholders.

The schema defines the shape. The template defines the layout. The engine connects them. The document assembles itself.
